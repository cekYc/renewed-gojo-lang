Kanıt 1

Hız (Performance)

Biz sadece hızlı değiliz — native hızdayız. Gojo bir sanal makine (VM) kullanmaz. Yazdığınız Gojo kodu, dünyanın en optimize derleyicisi olan Rust/LLVM altyapısına çevrilerek doğrudan native binary üretilir.

Yaptığımız CPU-bound testlerde:

Go (GC Overhead) ~444 ms
Gojo (No GC) ~259 ms

Bu fark bir optimizasyon numarası değil; GC’siz bellek modeli ve saf fonksiyon analizinin doğal sonucudur.

// Gojo Benchmark
fn fib(n: i64) -> i64 {
  if n <= 1 { return n; }
  return fib(n-1) + fib(n-2);
}
// Leke Analizi (Compile-Time)
fn login(user: Untrusted) {

  // ❌ HATA: Derleyici reddeder
  Database.query(user);

  validate user {
    success: {
      // ✅ user artık güvenli
      Database.query(user);
    }
  }
}
Kanıt 2

Leke Analizi (Taint Analysis)

Diğer dillerde şu soruyu sen sorarsın: “Bu veri güvenli mi?”
Gojo’da bu soruyu derleyici sorar.

Gojo’nun Leke Analizi motoru:

  • Dış dünyadan gelen veriyi Untrusted olarak işaretler.
  • Bu veri validate edilmeden DB’ye gidemez, Network’e çıkamaz.

Bu bir uyarı değildir. Derleme hatasıdır.
Gojo ile yazılmış bir kodda “Yanlışlıkla güvensiz kod yazdım” deme ihtimali yoktur.

Kanıt 3

Akıllı Motor (Smart Codegen)

Gojo derleyicisi, kodunuzu anlar.

  • Saf Matematik (Deterministic): Async overhead tamamen kaldırılır. Senkron, saf makine kodu üretilir.
  • I/O İşlemi (Nondeterministic): Otomatik olarak async/await moduna geçilir. Sen async yazmakla uğraşmazsın, Gojo senin yerine karar verir.

Geliştirici isterse bu davranışı açıkça kontrol edebilir. Ama zorunda değildir. Async yazmak zorunda kalmadan async çalışmak, Gojo’nun temel hedeflerinden biridir.

// Smart Codegen Örneği
deterministic fn mat() {
  // -> No Async Overhead
  return 2 + 2;
}

nondeterministic fn api() {
  // -> Auto Async/Await
  Http.get("url");
}

Sistem Mimarisi ve Detaylar

01. Sözdizimi (Syntax)

Mevcut Durum (Var)

  • Değişkenler: let x = 10 ile tanımlanır. Şu an hepsi "Immutable" gibi davranıyor, Rust tarafında mut olarak üretiyoruz. Tekrar atama mevcut.
  • Tipler: i64 (Tamsayı), String (Metin), Untrusted (Kirli Veri), Void, Array<T>.
  • Kontrol: if/else, while ve aralıklı for (0..10) döngüsü mevcut.

Planlanan (Eksik)

  • Boolean: true/false literal var ama tip olarak i64 (0/1) kullanıyoruz.
  • Float: Yok. Bilimsel hesaplama için şart.
  • Mutation: liste[0] = 5 diyemiyoruz. Bunu yaparsak tam bir dil oluruz.

02. Bellek & Scope

Bellek Yönetimi

GC Yok: Garbage Collector kullanmıyoruz. Değişken scope'un sonuna (}) geldiği an Rust'ın Drop mekanizması devreye girer. Bu yüzden 259ms hızındayız.

Manuel Yok: malloc veya free yok. Derleyici halleder.

Zombie Process Engelleme

Mevcut Hile: codegen.rs içinde scope sonuna sleep(50ms) koyarak "işlerin bitmesini bekleme" taklidi yapıyoruz.

Gerçek Çözüm: Rust'ın tokio::JoinSet yapısını entegre edeceğiz. Scope kapanmadan önce await_all() denilerek tüm işçilerin eve dönmesi beklenecek.

Validate Mekanizması

Untrusted Tipi: Dışarıdan gelen veri Untrusted tipindedir. src/analysis/taint.rs, kod derlenmeden tüm akışı tarar. Eğer Untrusted bir veri, validate edilmeden hassas bir yere giderse derlemeyi reddeder. Validate bloğu içinde değişken String (Trusted) olur.

Planlanan: Şu an validate "Her şeye tamam" diyor. İleride Regex/Schema eklenecek.

Hata Yönetimi

Mevcut: "Fail Fast" veya Varsayılan Değer. unwrap_or(0) kullanıyoruz.

Planlanan: Result<i64, Error> tipi ve match yapısı ile hatanın yönetilmesi.
